# Subtractive Design Philosophy — Cheat Sheet

## What It Is

A framework for building products that feel inevitable by removing everything that doesn't earn its place. Design by subtraction, not addition.

## The 6-Point Checklist

Before shipping any feature, screen, or element, answer these:

### 1. Lightness Test
> "Does this make the user's life feel lighter?"

Not "Is this useful?" but "Does this actively reduce friction or cognitive load?" If the answer is no, cut it.

### 2. Subtraction Audit
> "What can we remove without the product getting worse?"

Go through every button, menu item, configuration option. If removing it doesn't break the core workflow, remove it.

### 3. Earning Test
> "Does every element earn its place?"

"It might be useful to someone" is not justification. Design for your primary user, ruthlessly. Everything else is noise.

### 4. Whitespace Test
> "Is the empty space being used intentionally?"

Whitespace isn't wasted space. It's breathing room. If your interface is densely packed, you haven't subtracted far enough.

### 5. Honest vs. Impressive Filter
> "Are we shipping the honest version or the impressive version?"

The impressive version has more features, more options, more complexity. The honest version has exactly what users need. Impressive versions ship broken; honest versions ship inevitable.

### 6. Dieter Rams Gut Check
> "Would Dieter Rams approve?"

If the godfather of reductive design would frown at your interface, subtract more.

## 3-Phase Implementation

### Phase 1: Inventory
List every element in your product. Every button, every label, every animation, every config option.

### Phase 2: Score
Score each element 1-5 on "earns its place." Anything below 3 is a candidate for removal.

### Phase 3: Remove and Test
Remove the lowest-scoring elements. Test with users. If nobody notices the removal, it was right. If someone complains, evaluate whether they're the primary user.

## Key Principle

**Addition is visible. Subtraction is invisible.** The best design decisions are the ones users never notice because the thing that would have confused them was never there.
